# 9. AI 编程代理高效提示词模式

> 从本项目 132 次提交的实战中总结：哪些提示词模式让 AI 代理产出最好的结果。

---

## 核心原则

**AI 代理不是一次性问答机器，而是需要「驱动」的协作者。** 好的提示词模式能让代理：
- 持续工作而非浅尝辄止
- 自我纠错而非等待指令
- 深入挖掘而非表面应付

---

## 模式 1：循环收敛（最有效）

### 句式
```
多轮做 X 直至没问题
反复 X 直至没有新发现
多轮迭代 X，直至没有问题可以补充
```

### 为什么有效
- **设定了终止条件**而非固定次数——代理会自己判断何时够了
- **隐含了自检要求**——每轮都需要对比上一轮是否有改进
- **防止了"一轮就停"**——大部分代理默认做一轮就汇报，这个模式要求继续

### 实际效果（本项目数据）

| 提示 | 轮次 | 发现的问题 |
|------|------|-----------|
| "多轮审查直至没问题" | 9 轮 | 修正了 60+ 处错误 |
| "多轮从二进制反复核实" | 6 轮 | 发现 Machine ID 采集、782 遥测事件 |
| "多轮反复核实完善各个 CLI Agent 的内置命令" | 4 轮 | Aider 6→42 命令，Gemini 43 命令确认 |

### 变体
```
再次反复多轮审查你的修改直至没问题     # 强调"再次"防止代理认为上次已完成
多轮分析反编译二进制深度分析直至没有问题可以补充   # 指定方法（反编译）
不开源的 Code Agent 要再次多轮从二进制反复核实    # 指定范围（闭源工具）
```

## 模式 2：质量锚定

### 句式
```
要有深度，要达到 X 级别
会有其他模型来复查，务必保证质量
面向 X 开发者的深度
```

### 为什么有效
- **设定了质量基准**——"达到开发者级别"比"详细一点"清晰 100 倍
- **引入了外部审查压力**——"其他模型来复查"让代理更谨慎
- **避免了"差不多就行"心态**

### 实际效果

| 提示 | 产出变化 |
|------|---------|
| "要有深度" | /review 对比从 146 行重写为 318 行 |
| "会有其他模型来审核" | 触发了二进制反编译而非仅表面 strings 分析 |
| "达到 Code Agent 开发者对比功能的程度" | 从"有没有"对比升级为源码逐行分析 |

### 变体
```
深度都要到达 A 级别                    # 用评级锚定
证据要充分，内容要丰富，要有深度        # 多维度要求
要求分析源码或者二进制反编译分析        # 指定分析方法
```

## 模式 3：范围扩展

### 句式
```
其他的 X 是不是也可能有类似的缺失？
所有的 Agent 和所有命令都要有这个深度
每种 Code Agent 都要有 X
```

### 为什么有效
- **从一个案例推广到全部**——发现一个工具的问题后，自然推测其他工具也有
- **防止了选择性深入**——代理倾向于只深挖最熟悉的工具
- **建立了均衡标准**——所有工具同等对待

### 实际效果

| 提示 | 发现 |
|------|------|
| "其他 Code Agent 是不是也可能有类似的缺失" | Goose 缺失全部 16 命令，OpenCode 工具列表错误 |
| "所有的 Agent 和所有命令都要有这个深度" | 触发了 5 个工具的命令文档完整重写 |
| "开源的也要通过源码补齐同样的维度分析" | 7 个开源工具的遥测/安全分析 |

---

## 模式 4：证据要求

### 句式
```
记录证据
证据要充分
从二进制反复核实，并且记录证据
要从网络文档核实
```

### 为什么有效
- **防止了编造**——要求证据意味着代理必须执行实际的验证命令
- **创造了可审计的链条**——EVIDENCE.md 文件让后续审查有据可查
- **区分了"我知道"和"我确认"**——代理可能"知道"某工具有 X 功能，但要求证据后发现实际不存在

### 实际效果

| 提示 | 发现 |
|------|------|
| "记录证据" | 创建了 9 个 EVIDENCE.md（2083 行） |
| "从二进制反复核实" | Codex CLI `granular` 模式不存在（二进制返回 error） |
| "也要从网络文档核实" | Codex CLI 官方文档已添加 `granular`（二进制和网络矛盾） |

---

## 模式 5：思考引导

### 句式
```
思考：如何进一步完善？
思考：如何采集更多有用的信息？
思考，其他的 X 是不是也可能有类似的缺失？
```

### 为什么有效
- **先规划后执行**——防止代理直接冲进去写代码
- **让代理暴露思路**——你可以在执行前调整方向
- **产出结构化分析**——通常会输出优先级表格

### 实际效果

| 提示 | 产出 |
|------|------|
| "思考：如何进一步完善" | 产出了分优先级的改进清单（P0/P1/P2） |
| "思考：如何采集更多有用的信息" | 识别了 strace、依赖链审计等新分析维度 |

---

## 模式 6：及时提交

### 句式
```
工作要及时提交
做完 X 后 push
```

### 为什么有效
- **创建了检查点**——每次提交都是可回退的安全点
- **防止了"大爆炸"提交**——小步提交更容易审查
- **让进度可见**——提交历史就是工作日志

---

## 模式 7：结构指令

### 句式
```
每种 Code Agent 都可以单独一个目录，然后多个文件
文档的标题要有序号，方便阅读和引用
单独一页比较
```

### 为什么有效
- **明确了产出格式**——代理不用猜你想要什么结构
- **防止了信息堆积**——"单独一页"比"加到现有文档"更清晰
- **建立了可导航的体系**——序号让跨文档引用成为可能

---

## 模式 8：具体追问

### 句式
```
新加的 /review 和 /btw 没有？
比如最低支持 Node.js 版本这些
比如 Mac 地址等等
从文档看
要看源码啊，你要看仔细了
```

### 为什么有效
- **指出了具体遗漏**——比"再检查一下"有效 10 倍
- **给出了期望的精度**——"Mac 地址"告诉代理要查到硬件指纹级别
- **纠正了偷懒行为**——"要看源码啊"阻止代理只看文档不看代码

---

## 反模式（效果不好的提示）

| 反模式 | 问题 | 改进 |
|--------|------|------|
| "帮我写个文档" | 太模糊，产出泛泛而谈 | "写一个 Claude Code 用户指南，要有深度，同时易于阅读" |
| "检查一下" | 不知道检查什么 | "多轮审查直至没问题" |
| "详细一点" | 没有基准 | "要达到 Code Agent 开发者对比功能的程度" |
| "做完了吗" | 是/否问题 | "思考：还有哪些需要完善" |
| "继续" | 没有方向 | "继续，重点补充 X 维度" |

## 组合使用（最强效果）

本项目中效果最好的提示都是**多个模式的组合**：

```
不开源的 Code Agent 的内容，要再次多轮要从二进制反复核实完善，
并且记录证据，证据要充分，内容要丰富，要有深度，也要从网络文档核实
```

这一句包含了：
- **循环收敛**：再次多轮...直至
- **证据要求**：记录证据，证据要充分
- **质量锚定**：内容要丰富，要有深度
- **方法指定**：从二进制反复核实 + 从网络文档核实
- **范围限定**：不开源的 Code Agent

## 模式 9：自验证指令（最高杠杆）

### 句式
```
实现 X。测试用例：A→true，B→false。实现后运行测试。
完成后截图对比原图，列出差异并修复。
写完后运行 pnpm lint，修复所有错误。
```

### 为什么有效
- **Anthropic 官方推荐**，质量提升 2-3 倍
- 给代理一个**闭环反馈**——不依赖你来判断是否正确
- 代理可以自行迭代直到通过验证

### 关键原则
给代理一个检查自身工作的方法：测试命令、linter 输出、截图对比、类型检查。**没有验证方法的提示 = 盲人摸象。**

## 模式 10：探索-规划-实现-提交（EPIC 模式）

### 句式
```
1. 先读取 src/auth/ 理解现有架构（不要改代码）
2. 制定一个实施计划，列出要改的文件
3. 我确认后再开始实现
4. 实现后运行测试套件，修复失败
5. 提交并创建 PR
```

### 为什么有效
- 把**思考和执行分离**——避免解决错误的问题
- 每个阶段有明确的交接点（你可以修正方向）
- 经验法则：**一句话能描述 diff 就跳过规划，否则先规划**

---

## 模式 11：采访模式（Interview）

### 句式
```
我想做 X。在开始写代码之前，先用 AskUserQuestion 详细采访我。
问我技术实现、UI/UX、边界条件、顾虑和权衡。
不要问显而易见的问题——挖掘我可能没考虑到的难点。
持续采访直到覆盖所有方面，然后写一份完整的规格文档到 SPEC.md。
```

### 为什么有效
- 暴露你**自己都没想到的边界条件**
- 代理会问出"如果用户在中间断开连接怎么办？"这类问题
- 产出的 SPEC.md 可以在**新会话**中用干净上下文执行

---

## 模式 12：写手/审查者分离

### 句式
```
# 会话 A（写手）
实现 API 限流中间件

# 会话 B（审查者，新开）
审查 @src/middleware/rateLimiter.ts，检查边界条件和竞态

# 会话 A（修复）
以下是审查反馈：[粘贴 B 的输出]。请修复。
```

### 为什么有效
- 新会话没有"我刚写的代码"偏见
- 模拟真实的人类代码审查流程
- 审查质量显著高于自审（同会话内 /review）

---

## 模式 13：激进上下文卫生

### 句式
```
/clear                              # 任务间清理
/compact 保留 API 变更相关的讨论     # 有方向的压缩
```

### 关键规则
- **修正 3 次仍不对？** `/clear` + 重写提示。不要继续在污染的上下文里挣扎
- **任务切换时必须 `/clear`**。混杂不相关任务是最常见的效率杀手
- 用子代理调查，结果浓缩回主会话（保持主上下文干净）

> **社区共识：** 一个干净会话 + 好提示 ≫ 长会话 + 反复修正

## 模式 14：示例驱动

### 句式
```
参考 @src/widgets/HotDogWidget.tsx 的实现模式，创建 CalendarWidget。
看一下 tests/auth.test.ts 的写法，用同样的模式给 user.ts 写测试。
```

### 为什么有效
- **一个具体示例胜过十段描述**
- 代理会读取示例文件，理解命名规范、错误处理模式、测试结构
- 减少来回修正（代理直接匹配你期望的风格）

---

## 模式 15：症状 + 位置 + 成功标准

### 对比
```
❌ 修复登录 bug

✅ 用户报告 session 过期后登录失败。
   检查 src/auth/ 的 token 刷新流程。
   写一个能复现问题的失败测试，然后修复它。
```

### 为什么有效
- **症状**告诉代理问题是什么
- **位置**缩小搜索范围（不用搜索整个代码库）
- **成功标准**告诉代理何时算完成（测试通过）

---

## 模式 16："你确定吗？" 重新思考触发

### 句式
```
你确定这个实现是线程安全的吗？
这个方案会不会在高并发下有问题？再想想。
```

### 为什么有效
- 简单但出奇地有效——触发模型**重新审视**自己的答案
- 对于过于"自信"的第一反应特别有用
- 经常能发现代理自己都没注意到的边界条件

## 反模式补充

| 反模式 | 问题 | 改进 |
|--------|------|------|
| **厨房水槽会话** | 混杂不相关任务污染上下文 | `/clear` 分隔任务 |
| **修正 3 次以上** | 上下文充满失败尝试 | `/clear` + 更好的初始提示 |
| **CLAUDE.md 过长** | 规则被噪音淹没 | 保持 <200 行，无效规则删除 |
| **只信不查** | 看似正确但有隐藏边界条件 | 始终提供验证方法 |
| **无限探索** | 不定范围的"调查"耗尽上下文 | 缩小范围或用子代理 |
| **接受第一方案** | 第一个答案可能不是最优 | "给我 3 种替代方案"（非平凡问题） |
| **氛围编程** | 开放式需求无规格 | 先 SPEC.md 再编码 |

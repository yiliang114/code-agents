# Code Agent 架构

本文档解释 AI 编程代理的工作原理。

## 核心组件

### 1. 代理循环

大多数编码代理遵循类似的模式：

```
┌─────────────────────────────────────────────────────────────┐
│                     代理循环                                  │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐             │
│   │  输入    │ -> │  推理    │ -> │  行动    │             │
│   │(prompt)  │    │  (LLM)   │    │(工具调用)│             │
│   └──────────┘    └──────────┘    └──────────┘             │
│                                              │              │
│                                   ┌──────────▼──────────┐   │
│                                   │   观察结果          │   │
│                                   └──────────┬──────────┘   │
│                                              │              │
│                                   ┌──────────▼──────────┐   │
│                                   │   更新上下文        │   │
│                                   └──────────┬──────────┘   │
│                                              │              │
│                                              └──────┐       │

│                                              │  重复/完成   ││
│                                              └─────────────┘│
└─────────────────────────────────────────────────────────────┘
```

### 2. 常见模式

#### ReAct（推理 + 行动）

使用者：Gemini CLI、mini-swe-agent

```
1. 思考："我需要先读取文件"
2. 行动：read_file("main.py")
3. 观察：（文件内容）
4. 思考："我看到第 42 行有 bug"
5. 行动：edit_file("main.py", ...)
6. 观察：（成功）
```

#### 计划-执行-反思

使用者：Claude Code（计划模式）、Cline

```
1. 计划：创建详细计划
2. 确认：获取用户批准
3. 执行：执行计划步骤
4. 反思：审查结果
5. 调整：修复任何问题
```

#### 工具调用

使用者：大多数现代代理

```
{
  "type": "tool_use",
  "tool": "bash",
  "input": {"command": "pytest tests/"}
}
```

### 3. 关键架构决策

#### 单代理 vs 多代理

| 类型 | 优点 | 缺点 | 示例 |
|------|------|------|----------|
| 单代理 | 简单、可预测 | 并行有限 | Aider、Gemini CLI |
| 主+子代理 | 专用代理分工 | 协调开销 | Claude Code、OpenCode |
| 多代理并行 | 可视化并行工作 | 资源消耗大 | Qwen Code（Tmux/iTerm2 后端） |

#### 单体 vs 复合

| 类型 | 描述 | 示例 |
|------|-------------|----------|
| 单体 | 单个 LLM 做所有事情 | Aider、mini-swe-agent |
| 复合 | 多个专门组件 | OpenHands、SWE-agent |

#### 同步 vs 异步

| 类型 | 优点 | 缺点 | 示例 |
|------|------|------|----------|
| 同步 | 可预测、易于调试 | 较慢 | 大多数 CLI 工具 |
| 异步 | 更快、并行 | 更复杂 | Warp、Claude Code（子代理）|

#### LLM 接入策略对比

| 策略 | 实现 | 优势 | 劣势 | 使用者 |
|------|------|------|------|--------|
| **统一 SDK** | Vercel AI SDK | 一次集成，多提供商 | 受 SDK 限制 | OpenCode |
| **LiteLLM** | Python 包装 100+ 模型 | 模型最多 | Python 限定 | Aider, SWE-agent, OpenHands |
| **独立实现** | 每个提供商一个 Generator | 灵活，深度定制 | 维护成本高 | Qwen Code, Gemini CLI |
| **Handler 工厂** | 每个提供商一个 Handler | 精细控制 | 代码量大 | Cline (48+), Continue (60+) |
| **Provider trait** | Rust trait 抽象 | 性能最佳 | Rust 门槛 | Goose (58+) |
| **工厂函数** | Python match/case | 简单直接 | 扩展性一般 | Kimi CLI |
| **单提供商** | 直连特定 API | 最深度优化 | 锁定 | Claude Code |

### 上下文管理策略对比（源码分析）

| 策略 | 工具 | 实现方式 |
|------|------|---------|
| **AST 仓库映射** | Aider | Tree-sitter 解析 30+ 语言，提取函数/类定义，SQLite 磁盘缓存 |
| **语义向量索引** | Continue | Tree-sitter AST + LanceDB 向量 + SQLite FTS5 全文索引 |
| **聊天压缩** | Gemini CLI, Qwen Code, OpenCode | LLM 摘要旧消息，Token 阈值触发 |
| **递归压缩** | OpenHands | Condenser 多深度递归摘要 |
| **ChatChunks 分块** | Aider | 系统/示例/仓库/文件/历史分块，独立缓存控制 |
| **30+ 上下文提供器** | Continue | 文件/Git/Jira/数据库等 30+ 来源动态注入 |
| **工具输出截断** | OpenCode | 32K token 上限，溢出自动存文件 |
| **Prompt 缓存** | Aider, Gemini CLI | Anthropic cache_control + 后台保活 ping |

### Token 预算分配

大多数代理这样分配 token：

```
总计：200,000 tokens
├── 50,000：系统提示 + 指令
├── 100,000：仓库上下文
├── 30,000：最近对话历史
└── 20,000：当前任务 + 工具
```

## 实现示例

### mini-swe-agent（100 行）

```python
class SimpleAgent:
    def __init__(self, model):
        self.model = model
        self.history = []

    def run(self, task):
        while True:
            # 从 LLM 获取下一个行动
            response = self.model.predict(
                task,
                self.history
            )

            # 执行行动
            result = self.execute(response.action)

            # 更新历史
            self.history.append((
                response.action,
                result
            ))

            # 检查是否完成
            if response.done:
                break
```

这是更复杂代理构建的基础。

## 3. Codex

> **193 merged PR**——OpenAI 内部高密度节奏，单 PR 粒度极细（常见 +0/-X 拆分式重构）。主题归类：

### 🔥 Guardian 行动审批守门（jif-oai 主力，本周最大投入）

**先澄清：Guardian 不是「PR 代码评审」，而是 Codex 的「行动审批守门员」**——agent 每次准备跑命令 / 写文件 / 删除 / 联网之前，先把这个动作交给一个独立的 Guardian LLM 评一遍：这事**有多危险**（low/medium/high/critical）+ **用户到底授权了没**（high/medium/low/unknown），再决定放行 / 追问 / 拒绝。源码是 `codex-rs/core/src/guardian/` 一整个独立模块（专属 prompt + policy 模板）。

可以理解成：**比"规则匹配式权限弹窗"更聪明的一层——它会先用只读工具取证再判**（比如删文件前先 `ls` 看目标是不是空的，联网前先看 git remote 判断仓库是不是公开的），还专门防 prompt injection（把工具输出当"不可信证据"，不被里面"求放行"的话术带跑）。

本周做的是给这个系统**省钱 + 装仪表盘**：

- **省钱（prompt cache 复用）**（#24803/#24891/#24893/#24894/#24895）—— 之前 fork 出来的 review session 各用各的 cache key，本该共享缓存的却没共享，白烧 token；改成稳定 key 后命中率上去了
- **装仪表盘（metrics）**（#24897）—— 加 OpenTelemetry 指标（review 次数 / 延迟 / token 消耗 / 最终是放行还是拒绝），方便灰度时盯着 Guardian 的行为按模型、动作类型、失败原因分类对比
- **修隔离**（#24714）—— auto-review 线程不该继承父会话的 notify hook，清掉防泄漏

## 4. Claude Code（闭源，按 release notes v2.1.152 ~ v2.1.158）

> Claude Code 不开源，看不到 PR；以下按官方 changelog 梳理。本周一口气发了 **7 个版本**，最大事件是 05-28 的模型 + 编排能力大版本。

### 🎉 重大：Opus 4.8 + dynamic workflows（v2.1.154, 05-28）

- **Opus 4.8 发布**，默认 high effort，最难的活用 `/effort xhigh`；fast mode 成本砍到原来一小部分（2 倍单价换 2.5 倍速度）
- **dynamic workflows 上线**：让 Claude 即兴写编排脚本、后台 fan-out 几十到上百 agent，`/workflows` 看运行——详见 [Claude Code Dynamic Workflows Deep-Dive](../claude-code-dynamic-workflows-deep-dive.md)
- 配套：lean system prompt 成为除 Haiku/Sonnet/Opus 4.7 及更早外的默认；少问没必要的多选题（够上下文就直接干）

### Auto mode 从"需同意"转向默认

- **Auto mode 不再要求 opt-in consent**（v2.1.152）→ 进而在 Bedrock / Vertex / Foundry 上对 Opus 4.7/4.8 可用（v2.1.158，env `CLAUDE_CODE_ENABLE_AUTO_MODE=1`）
- auto-mode 分类器**加强了数据外泄检测**，尤其是"整仓库内容批量外传"
- 这条线和 Codex Guardian、Qwen 的 AUTO classifier（#4376）是**同一战场**：让 agent 自动放行安全动作、只在真危险时才拦

### Plugin / Skill 体系成熟（本周第二大投入）

- `.claude/skills` 目录下的 plugin **自动加载，不再需要 marketplace**；`claude plugin init <name>` 脚手架
- `/plugin` 参数自动补全 + Discover 标 "suggested for this directory"
- `plugin.json` 支持 `defaultEnabled: false`（默认关、按需开）
- `/reload-skills` 不重启就重扫；`SessionStart` hook 可 `reloadSkills: true` 让 hook 装的 skill 当场生效
- skill / slash command frontmatter 可设 `disallowed-tools` 在 skill 活跃时临时摘掉某些工具
- 新 `MessageDisplay` hook（显示前改写/隐藏 assistant 文本）+ `SessionStart` 可设会话标题

### review / 代码清理命令

- `/code-review --fix` 现在 review 完**直接把建议应用到工作树**（reuse / 简化 / 效率）
- `/simplify` 拆出来只做"清理向 review"（不再跑完整找 bug 流程）

### `claude agents`（后台会话）+ worktree

- `claude agents` 里 `! <command>` 把 shell 命令当**可 attach/detach 的后台会话**跑（也可 `claude --bg --exec`）
- `EnterWorktree` 支持**会话中途切换** Claude 管理的 worktree；agent 干完自动解锁 worktree 方便 `git worktree remove`
- 一大批 background session 稳定性修复（resume 日期错乱 / 30 天清理孤儿 worktree / 切到正确 linked worktree 等）

## 6. 值得 Qwen Code 关注的点

1. **OpenCode 的 acp-next 换代**：OpenCode 把整个 ACP client 实现重写并 promote，与 Qwen daemon 的 ACP HTTP transport（#4472）形成 ACP 生态两端呼应——**ACP 正成为多家共识**，Qwen 押注 dual transport 方向得到旁证。
2. **Codex 的 Guardian 行动审批守门**：这其实对标的是 **Qwen 的 permission flow / PermissionMediator，不是 `/review`**。差别在于——Qwen 现在是"规则匹配 + 多 client 投票"决定要不要批一个动作；Guardian 是让一个 LLM **逐动作判风险**，还会先用只读工具取证（删文件前先看目标是不是空的）再决定放行/追问/拒绝，并显式防 prompt injection。如果 Qwen 想让权限审批从"硬规则"升级到"会思考、会取证"，Guardian 的思路值得参考；它本周还顺带做了 prompt cache 复用（省 token）+ OpenTelemetry 指标（可观测），这两点对任何 LLM 子系统都通用。
3. **Codex Python SDK 独立 beta**：Qwen 的 SDK 是 TypeScript 单一栈，Codex 已铺 Python SDK 产品线——多语言 SDK 是平台化必经路。
4. **四家都在收敛 compaction / context 策略**：Qwen 一周内两次重构（#4345 ladder → #4599 summary+restoration），Claude Code 也在调 thinking summary / lean system prompt 默认——说明 context 管理仍是核心未定型区域。
5. **Claude Code 的 dynamic workflows = Qwen daemon 还差的最上层抽象**：Claude 把"LLM 即兴写编排脚本 + 后台 fan-out 几十到上百 agent"做成正式能力（详 [deep-dive](../claude-code-dynamic-workflows-deep-dive.md)）。Qwen daemon 底层管道（non-blocking `/prompt` / ACP HTTP / MCP bridge）已就绪，缺的正是这层编排胶水，估 ~9-13 人周可补（详 [daemon README §零 未来方向参照](../qwen-code-daemon-design/README.md)）。
6. **「自动放行安全动作、只拦真危险」正成为四家共同战场**：Claude Auto mode（本周转默认 + 查外泄）/ Codex Guardian（LLM 取证式审批）/ Qwen AUTO classifier（#4376 PermissionDenied hooks）—— 三家都在用 LLM 把权限审批从"每次都问"推向"该问才问"，是 agent 走向更自主的共同方向。

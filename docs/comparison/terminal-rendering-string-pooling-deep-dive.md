# Qwen Code 改进建议 — 终端渲染字符串池化 (Terminal Rendering String Pooling)

> 核心洞察：CLI Agent 的终端用户界面（TUI）往往承担着极重的渲染任务——当 Agent 流式输出代码差异（Diff）或长文档时，React Ink（或类似的渲染引擎）需要以 60 帧的速度重绘终端。每一次重绘，底层都要经历数千个字符的 ANSI 转义、背景色判定和合并。如果每次都重新分配内存和对比字符串，会导致 JavaScript 的垃圾回收（GC）疯狂触发，进而造成肉眼可见的“屏幕闪烁”和击键卡顿。Claude Code 在其深度定制的终端引擎中引入了极致的“字符与样式池化（CharPool/StylePool）”机制，实现了零开销差异比对；而 Qwen Code 目前使用的是开箱即用的标准渲染组件。
>
> 返回 [改进建议总览](./qwen-code-improvement-report.md)

## 一、为什么高频打印会导致输入卡顿？

### 1. Qwen Code 现状：字符串灾难与 GC 尖峰
当大模型在终端以 30 token/s 的速度倾泻大段的 TypeScript 源码时，每收到一个字：
- 渲染引擎可能要把几百行的可见区域全部更新一遍。
- 每一个高亮的关键字，都伴随着一段冗长的 ANSI 转义字符：`\x1b[32mconst\x1b[0m`。
- **痛点**：在传统的 React/Node.js 实现中，每次计算屏幕变更，底层都要对比这两张由字符串拼接而成的巨大画布。如果字符串不同，底层库就会频繁分配并丢弃几十兆的内存对象。
- 用户在这个阶段如果试图在底部的输入框里敲字，键盘事件循环会被 GC（垃圾回收器）强行打断，导致长达几百毫秒的“键盘失灵”或跳字。

### 2. Claude Code 解决方案：指针映射与整数对比
Claude Code 为了打造极致流畅的控制台体验，手写了超过 7000 行的 `ink/` 终端渲染包。
其中最核心的性能防线就是：**对象池化 (String Interning / Pooling)**。

在 `ink/screen.ts` 和 `ink/output.ts` 的深处：
它构建了 `CharPool`、`StylePool` 和 `HyperlinkPool`。

#### 机制一：字符串 -> 整数的全局映射
当系统遇到一个紫色的背景样式时，它不存储冗长的配置对象或 ANSI 字符串。
它会在 `StylePool` 中查表，把这个样式注册为一个唯一的 `number`（比如 `StyleID = 42`）。

#### 机制二：画布变成了轻量的数字矩阵
整个终端屏幕（假如是 100 列 × 50 行）不再是由字符串构成的巨型网格，而是变成了两个非常紧凑的 `Uint32Array` 或者数值矩阵：
- 一个存放文字的 `CharID`。
- 一个存放样式的 `StyleID`。

#### 机制三：O(1) 复杂度的帧间 Diff
当下一帧到来时，引擎要判断某一个格子是否需要向终端下发指令更新。
**以前**：深拷贝两个复杂的 Style 对象去比较。
**现在**：`if (prevFrame.style[i] !== nextFrame.style[i])`，仅仅是一次微秒级的 CPU 原生整型数字比较 (O(1))！

不仅消灭了成千上万次无效的字符串比较，由于 ID 是被长期复用的（池化），整个渲染循环几乎**不产生任何临时的生命周期对象**，GC 完全进入了休眠状态。

## 二、Qwen Code 的改进路径 (P2 优先级)

对于追求极致丝滑手感的代码编辑器/终端工具而言，渲染重绘是必须翻越的大山。

### 阶段 1：剥离昂贵的重绘库
1. 分析目前 Qwen Code 在终端中打印 `Markdown` 源码高亮的部分。
2. 避免使用那些“每渲染一行就抛出成百上千个小组件”的低效 UI 库。尽量将静态代码输出转为流式的块追加。

### 阶段 2：引入缓存池机制 (针对状态更新组件)
1. 在渲染频繁更新的区域（例如最底部的 Status Bar 或 Spinner），可以手写一个非常迷你的 `StylePool`。
2. 记录上一次刷给终端特定游标位置的 ANSI Style 签名。如果这轮循环中样式签名不变，仅更新内部文本字符。

### 阶段 3：配合双缓冲与批量输出
将池化对比技术与 `DEC 2026`（同步屏幕更新特性）结合起来。先在内存矩阵中进行整型比较，计算出最小脏矩形（Dirty Rectangle），然后打包成一段单一的 Buffer，通过 `process.stdout.write` 一次性全量刷入终端。

## 三、改进收益评估
- **实现成本**：极高（如果全量重写 TUI 引擎的话）。对于 Qwen 而言，建议仅在特定的高频渲染流（如 Markdown 组件）中局部应用对象复用技术，这属于中期优化规划。
- **直接收益**：
  1. **告别高频输出时的电脑风扇狂转**：CPU 消耗直线下降，避免 Agent 被误认为是挖矿木马。
  2. **坚若磐石的输入响应**：即使在模型以极限速度刷屏的时候，用户在命令框中敲击退格键和上下方向键，依然保持绝对的 60fps 响应速度。
